## 1. 목표 및 성공 기준

### 목표

home_agent v0.1을 **본인이 매일 1주일 이상 자원해서 쓸 수 있는 상태**로 만든다. 일정·TODO·가계부 3개 도메인을 자연어 챗 + 캘린더/대시보드로 사용 가능하게 한다.

### 성공 기준 (Done의 정의)

v0.1 종료 시점(4주차 끝)에 다음 5가지가 매일 자연스럽게 일어나야 한다:

1. **자연어 입력** — 챗에서 일정·TODO·가계 입력이 자연어로 가능 ("내일 3시 회의 잡아줘", "어제 점심 만원")
2. **Google Calendar 통합** — 일정은 Google Calendar에서 가져온 것 + 자체 입력한 것이 캘린더 화면에 함께 보인다 (Google = 읽기 전용 source)
3. **Todoist 양방향 동기화** — TODO는 Todoist와 양방향 동기화되어 home_agent 입력이 Todoist 앱에도 즉시 반영된다
4. **카테고리 자동 추정** — 가계부 거래는 챗으로만 입력해도 카테고리 자동 추정이 80% 이상 맞다 (감각적 기준)
5. **아침 다이제스트** — 매일 아침 07시 다이제스트(인앱 메시지)로 오늘 일정/마감 임박/예산 상태가 한 화면에 뜬다

### 비목표 (v0.1에서 명시적으로 제외)

가족 공유, 모바일 접근, 영수증 OCR, 자산/재무 에이전트, 아이디어/파일 에이전트, 음성 입력, 외부 알림(이메일/푸시), 데스크톱 앱 래핑(Tauri/Electron), VPS 호스팅.

## 3. 빌드 단계 (4주차 계획 — Approach A: 가로로 얇게)

### Week 1 — 일정 도그푸딩 활성화

- backend: `/chat` 엔드포인트, orchestrator stub, schedule_agent + calendar MCP, Google Calendar OAuth + 읽기 동기화 (cron 15분마다)
- frontend: 챗 페이지, 캘린더 페이지 (Google + 자체 일정 통합 표시)
- db: `schedule.events`, `core.sessions`, `core.messages` 마이그레이션
- **Week 1 종료 기준**: 챗에서 "내일 3시 회의 잡아줘" → 일정 생성, 캘린더에서 확인. Google Cal의 기존 일정도 보임.

### Week 2 — TODO + Todoist 양방향

- backend: todo_agent + todo MCP, Todoist OAuth, 양방향 sync 정책 (last-write-wins + Todoist source on conflict)
- db: `todo.tasks` 마이그레이션, `external_id`/`synced_at` 필드 포함
- frontend: 대시보드 페이지 신규 (오늘 일정 + 임박 TODO 위젯)
- **Week 2 종료 기준**: home_agent에서 TODO 추가 → Todoist 앱에 즉시 반영. Todoist에서 완료한 작업도 home_agent에 반영.

### Week 3 — 가계부

- backend: ledger_agent + ledger MCP, `category_inference` LLM 도구 (Haiku tier)
- db: `ledger.transactions`, `ledger.categories`, `ledger.budgets` 마이그레이션
- frontend: 대시보드에 어제 지출 위젯 추가
- **Week 3 종료 기준**: "어제 점심 만원 썼어" → 자동 카테고리=식비 추정, ledger에 저장. 대시보드에서 누적 확인.

### Week 4 — 아침 다이제스트 + 마감

- backend: APScheduler job (매일 07시 다이제스트 생성 → `core.messages`에 system 메시지로 push)
- frontend: 대시보드 상단에 오늘 다이제스트 강조
- 테스트 정비, 도그푸딩 시작
- **Week 4 종료 기준**: 1주일 도그푸딩 검증, 매일 자원해서 쓰는 상태 도달.

## 4. 컴포넌트 / 아키텍처

`planning/AGENTS.md`의 구조를 유지하되 v0.1 범위로 축소.

```
                            ┌─────────────────┐
                            │  React (Vite)   │
                            │  Chat / Dashboard / Calendar │
                            └────────┬────────┘
                                     │ HTTP + SSE
                                     ▼
                            ┌─────────────────┐
                            │   FastAPI       │
                            │   /chat /events │
                            └────────┬────────┘
                                     │
                            ┌────────▼────────────────────────┐
                            │   Orchestrator (Claude SDK)     │
                            │   intent → route → aggregate    │
                            └─────┬───────┬───────┬───────────┘
                                  ▼       ▼       ▼
                          schedule    todo    ledger
                            agent    agent    agent
                                  │       │       │
                                  ▼       ▼       ▼
                            MCP calendar  todo  ledger
                                  │       │       │
                            Google Cal  Todoist  (자체)
                                  │       │       │
                                  ▼       ▼       ▼
                            ┌────────────────────────┐
                            │  Postgres 16 + pgvector │
                            │  schedule.* todo.* ledger.* core.* memory.* │
                            └────────────────────────┘

                            ┌─────────────────────────┐
                            │  APScheduler            │
                            │  - 07시 다이제스트       │
                            │  - 15분마다 Google sync │
                            │  - Todoist sync         │
                            └─────────────────────────┘
```

### 책임 분리

- **Orchestrator**는 자기 자신은 DB에 직접 접근하지 않는다. 서브에이전트 호출과 `memory.*` write만 한다.
- **서브에이전트**는 자기 도메인 MCP 서버의 도구만 호출. 다른 도메인 스키마 접근 금지.
- **MCP 서버**는 Postgres 도메인 스키마와 외부 API(Google/Todoist) 호출을 캡슐화.

## 5. 데이터 모델

각 도메인 스키마는 `planning/AGENTS.md`의 정의를 따른다. v0.1에서 추가/변경되는 필드만 명시:

### schedule.events

```sql
CREATE TABLE schedule.events (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES core.users(id),
  source      TEXT NOT NULL DEFAULT 'local',  -- 'local' | 'google'
  external_id TEXT,                            -- Google Calendar event id
  title       TEXT NOT NULL,
  start_at    TIMESTAMPTZ NOT NULL,
  end_at      TIMESTAMPTZ,
  description TEXT,
  synced_at   TIMESTAMPTZ,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX idx_events_external ON schedule.events(source, external_id)
  WHERE external_id IS NOT NULL;
```

### todo.tasks

```sql
CREATE TABLE todo.tasks (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       UUID NOT NULL REFERENCES core.users(id),
  source        TEXT NOT NULL DEFAULT 'local', -- 'local' | 'todoist'
  external_id   TEXT,                          -- Todoist task id
  title         TEXT NOT NULL,
  project       TEXT,
  due_at        TIMESTAMPTZ,
  completed_at  TIMESTAMPTZ,
  priority      SMALLINT,                      -- Todoist 1-4 매핑
  synced_at     TIMESTAMPTZ,
  sync_version  INTEGER NOT NULL DEFAULT 0,    -- conflict resolution
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX idx_tasks_external ON todo.tasks(source, external_id)
  WHERE external_id IS NOT NULL;
```

### ledger.transactions

```sql
CREATE TABLE ledger.transactions (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES core.users(id),
  occurred_on DATE NOT NULL,
  amount      NUMERIC(12, 2) NOT NULL,        -- 음수 = 지출, 양수 = 수입
  category    TEXT,
  category_confidence REAL,                    -- LLM 추정 신뢰도 0..1
  category_method TEXT,                        -- 'llm' | 'rule' | 'manual'
  memo        TEXT,
  account     TEXT,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### core.messages

기존 정의 그대로. `role` 컬럼은 `'user' | 'assistant' | 'tool' | 'system'`을 포함하며, 아침 다이제스트는 `role='system'`로 저장된다.

## 6. 외부 연동

### Google Calendar (읽기 전용)

- **OAuth 2.0** — Google Cloud Console에서 OAuth client 발급, 사용자 동의 후 refresh token을 `core.users.external_tokens` JSONB 컬럼에 저장.
- **동기화 주기** — APScheduler 15분 cron으로 `events().list(timeMin=now-1d, timeMax=now+90d)` 호출.
- **저장 방식** — `schedule.events.source='google'`로 upsert. external_id로 dedup.
- **삭제 처리** — Google 측에서 삭제된 일정은 local에서도 soft-delete (별도 컬럼) — v0.1에서는 단순화하여 마지막 동기화 시점 이전의 google 일정은 hard-delete 가능.
- **충돌** — 사용자가 home_agent에서 Google 동기화된 일정을 수정하려 하면 거부 (Google에서만 수정 가능). UI에서 명확히 표시.

### Todoist (양방향)

- **OAuth 2.0** — Todoist API token.
- **동기화 주기** — APScheduler 5분 cron + 사용자 액션 발생 시 즉시.
- **충돌 정책** — `sync_version` 컬럼을 사용한 last-write-wins. 양쪽 모두 변경된 경우 **Todoist를 source of truth**로 간주하고 home_agent 변경분 폐기. 이유: Todoist 모바일 앱에서의 입력이 더 빈번할 것으로 예상.
- **로컬 생성 흐름** — home_agent에서 task 추가 → DB에 `source='local', external_id=NULL`로 INSERT → 즉시 Todoist API 호출 → 응답의 task id로 `source='todoist', external_id=<id>` 업데이트.
- **삭제** — Todoist 측 삭제는 local에서도 hard-delete. Local에서의 삭제는 Todoist API delete 호출.

## 7. 에러 / 엣지케이스

### 자연어 모호성

- "오늘 점심 만원" — 날짜 명확, 금액 명확, 카테고리 미지정. → LLM이 `식비` 추정, `category_confidence < 0.8`이면 챗에서 사용자에게 확인 1회.
- "내일 회의" — 시각 미지정. → "몇 시쯤이 좋을까요?" 되묻기.
- 영수증/이미지 첨부 — v0.1 비목표. "v0.2에 추가 예정" 안내.

### 동기화 실패

- Google/Todoist API 호출 실패 시 `agent_runs.error`에 기록하고 다음 cron 주기에 재시도. 사용자에게 인앱 배너 표시.
- OAuth refresh token 만료 시 챗에서 "Google 재인증이 필요해요" 알림 + 설정 페이지 링크.

### 카테고리 추정 실패

- LLM이 카테고리를 모르면 `category=NULL, category_method='llm', category_confidence=0`으로 저장하고 가계부 화면에서 "미분류"로 표시. 사용자가 수동 분류 시 `category_method='manual'`로 갱신.

### 동시 입력 충돌

- 같은 일정/TODO에 대해 chat과 UI에서 동시 편집 시도 — v0.1에서는 last-write-wins (UI 시간 기준). 충돌 해결 UI 없음.

## 8. 테스트 전략

### 1차: 도그푸딩

본인이 매일 사용하면서 발견하는 마찰점이 1차 검증. `progress.txt`에 매일 한 줄씩 기록.

### 2차: 단위 + 통합 회귀

- **단위 테스트** (pytest) — 각 서브에이전트의 핵심 분기, category_inference 휴리스틱, 충돌 해결 로직.
- **통합 테스트** — `/chat` 엔드포인트가 mocked LLM과 함께 schedule_agent를 호출하고 DB에 일정이 저장되는 전체 흐름. testcontainers로 실제 Postgres 사용.
- **E2E** (Playwright, v0.1에 1-2개만) — 챗에서 일정 추가 → 캘린더 화면에서 확인하는 golden path 한 줄.
- **외부 API 테스트** — Google/Todoist 클라이언트는 VCR-style 녹화 응답 사용. 실제 API 호출은 nightly 한 번.

### 비활성화

- 부하 테스트, 보안 침투 테스트, 접근성 테스트 — v0.1 비목표. 1인 로컬 사용 가정.
